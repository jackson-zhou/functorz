# 事件流程编辑器 — 完善设计

## 1. 与上一版设计文档的关系

本文档是对 `2026-07-09-event-flow-editor-design.md` 的补充完善。上一版设计的核心功能（编辑器弹窗、节点面板、属性面板、数据持久化）已基本实现。本文档聚焦于：

- 条件节点规则配置 UI 的详细设计

- 节点间配置的联动关系
- 节点类型充分性校验（设计原则：无对应节点则先创建）

## 2. 节点类型清单

### 2.1 设计原则

与页面设计一致，事件流程编辑器中的每个节点（流程步骤）也遵循"组件化"原则：**如果流程图中需要使用某种节点类型，而该类型不存在，则必须先创建节点定义，再绘制流程图。**

### 2.2 已有节点类型（7 个）

| 节点 | 类型标识 | 形状 | 颜色 | 出边数 | 配置项 |
|------|----------|------|------|--------|--------|
| 开始 | `start` | 圆形 | 绿 #52c41a | 1 | 无 |
| 结束 | `end` | 圆形 | 绿 #52c41a | 0 | 无 |
| API 调用 | `api` | 圆角矩形 | 蓝 #1890ff | 1 | url, method(GET/POST) |
| 通知提示 | `alert` | 圆角矩形 | 黄 #faad14 | 1 | type(success/error/info), message |
| 页面跳转 | `navigate` | 圆角矩形 | 青 #13c2c2 | 1 | pageId |
| 条件分支 | `condition` | 菱形 | 紫 #722ed1 | 2(是/否) | logic(AND/OR), rules[] |
| **填充数据** | **`setData`** | **圆角矩形** | **橙 #fa8c16** | **1** | **target, source** |

### 2.3 流程需求校验

下面逐项检查演示流程是否能用已有节点完整表达：

| 流程2 忘记密码 | start, navigate, end | ✅ 全部已有 | 无需新建节点 |
| 流程3 注册账号 | start, navigate, end | ✅ 全部已有 | 无需新建节点 |
| 流程4 商品卡片 | start, api, condition, navigate, alert, end | ✅ 全部已有 | 无需新建节点 |
| 流程5 搜索栏 | start, condition, navigate, alert, end | ✅ 全部已有 | 无需新建节点 |
| 流程6 Banner | start, navigate, end | ✅ 全部已有 | 无需新建节点 |

## 3. 事件类型扩展

### 3.1 当前状态

`events` 字段仅支持 `tap`（点击事件），绑定在任意组件上：

```typescript
// 当前 schema
events: z.object({
  tap: flowSchema.optional(),
}).strict()
```

### 3.2 扩展方案

为支持首页加载流程，新增 `load` 事件类型。`load` 绑定在 **Page 根节点**上，页面打开时触发。

```typescript
// 扩展后
events: z.object({
  tap: flowSchema.optional(),
  load: flowSchema.optional(),
}).strict()
```

### 3.3 事件绑定规则

| 事件 | 绑定对象 | 触发时机 | 适用场景 |
|------|----------|----------|----------|
| `tap` | 任意组件（Button, Text, Image, ProductCard 等） | 用户点击组件 | 按钮点击、商品卡片点击、导航切换 |
| `load` | **仅 Page 根节点** | 页面加载完成 | 首页数据加载、初始化请求 |

### 3.4 UI 变更

在 App.tsx 右侧「交互」Tab 中，当选中组件为 Page 时，额外展示「页面加载事件」配置区：

```
选择 Page 时:
┌──────────────────────────────┐
│ 页面加载事件 (load)            │
│ ✓ 已绑定（4 个节点）           │
│ [编辑流程] [清除]             │
├──────────────────────────────┤
│ 点击事件 (tap)                │
│ 暂未绑定（Page 通常不含 tap）  │
└──────────────────────────────┘
```

## 4. 条件节点规则配置 UI

### 4.1 当前状态

FlowPropertyPanel 中 condition 节点的配置为：

```
┌──────────────────────────────┐
│ 条件逻辑                      │
│ [AND ▼]                      │
│ 条件规则配置暂未开放           │
└──────────────────────────────┘
```

### 4.2 目标 UI

```
┌──────────────────────────────────┐
│ 条件分支配置                      │
├──────────────────────────────────┤
│ 逻辑关系:                         │
│  ○ 全部满足 (AND)   ● 任一满足 (OR)│
│                                  │
│ 规则列表:                         │
│ ┌──────────────────────────────┐ │
│ │ 字段名: [response.code    ]  │ │
│ │ 运算符: [==       ▼]        │ │
│ │ 值:     [0                ]  │×│ │
│ ├──────────────────────────────┤ │
│ │ 字段名: [response.msg     ]  │ │
│ │ 运算符: [contains  ▼]        │ │
│ │ 值:     [成功             ]  │×│ │
│ └──────────────────────────────┘ │
│                                  │
│ [+ 添加规则]                      │
│                                  │
│ 分支说明:                         │
│  是 (right) → 条件满足时执行      │
│  否 (bottom) → 条件不满足时执行   │
└──────────────────────────────────┘
```

### 4.3 交互设计

| 操作 | 行为 |
|------|------|
| 切换 AND/OR | 更新 `config.logic`，实时生效 |
| 修改字段名/运算符/值 | 更新对应 rule 的 field/operator/value |
| 点击 × 删除规则 | 从 rules 数组中移除该条 |
| 点击「添加规则」 | 新增一条空规则 `{ field: '', operator: '==', value: '' }` |
| 最少规则数 | 1 条（不能全部删除） |

### 4.4 运算符选项

| 运算符 | 含义 | 适用场景 |
|--------|------|----------|
| `==` | 等于 | 状态码判断 |
| `!=` | 不等于 | 错误判断 |
| `>` | 大于 | 数值比较 |
| `<` | 小于 | 数值比较 |
| `>=` | 大于等于 | 阈值判断 |
| `<=` | 小于等于 | 阈值判断 |
| `contains` | 包含 | 字符串匹配 |

### 4.5 数据模型（已有，无需修改）

```typescript
// 已在 schema 中定义
interface ConditionRule {
  field: string       // 字段名，如 "response.code"
  operator: '==' | '!=' | '>' | '<' | '>=' | '<=' | 'contains'
  value: string       // 比较值，如 "0"
}

interface ConditionConfig {
  rules: ConditionRule[]   // 规则列表
  logic: 'and' | 'or'      // 逻辑关系
}
```

### 4.6 实现要点

- 修改 `FlowPropertyPanel.tsx` 中 `case 'condition'` 分支
- 规则列表使用受控组件，每次修改调用 `onConfigChange`
- 空规则（field 为空）视为占位，保存时不过滤（保留用户意图）
- 不涉及 schema 修改（数据结构已完备）
