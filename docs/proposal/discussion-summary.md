## 1. 项目背景与目标

- 甲方要把其 **PC 版低代码平台迁移到微信小程序 / H5**。
- 这是多期、多年的大工程：甲方 PC 版由一个团队做了 **3 年多**，量级约 ¥300–500 万投入；完整平台价值属**七位数级别**。
- 承接策略：**分期交付**。第一期做「走通式骨架 + 可演示的拖拉搭建」当甜头，让甲方看到方向没错、且承接方能搞定。
- 现有代码基（functorz）已有可视化搭建器、组件注册表、schema、runtime-renderer，可作为 **E1 的起点**。

## 2. 核心架构：三大引擎（承接方自己的心智模型，主线）

- **E1 界面布局引擎**：渲染 UI，跑在端上（H5 + 小程序，Taro）。现有代码的延伸。
- **E2 图化执行引擎 + 可扩充算子**：事件触发的流程编排；由算子按顺序 / 分支 / 循环串起来。
- **E3 服务引擎**：实现「重算子」（数据、密钥、三方、不可信计算）。
- **RPC 算子 = E2 → E3 唯一接缝**：E2 只需会「跑本地算子」+「发 RPC」，重活全甩给 E3。

> 与「9 部分」分解的对应：三大引擎 = 运行时；描述层（组件 / 数据 / 逻辑 / 绑定）是喂给引擎的料；搭建器、发布托管、平台底座是正交能力。

## 3. 关键架构原则

- **闭包性质**：搭建器能表达的，运行时必须能解释。保持描述声明式、运行时是解释器 → 靠「扩词汇」（加组件 / 算子 / 字段类型）成长，不必重写引擎。
- **解释而非 eval**：目标含小程序，运行时不能 `eval`。表达式与流程要么数据化解释、要么编译成可移植代码。这条既是架构约束，也是过微信审核的关键。
- **算子上不上服务端，判据是「能力边界」不是「复杂度」**：需要数据 / 密钥 / 三方 / 不可信计算的走 E3；纯本地的（setState / 条件 / 导航 / 格式化）留 E2。
- **状态存储**：E1↔E2 的共享内核（不是引擎，是共享内存）。至少两层作用域：**页面态 + 会话态**（token 之类要跨页面活）。
- **E2 运行位置**：设想为「一套可移植解释器」——UI 触发的流程在端上（贴着状态），headless 流程（定时 / webhook / 保存前钩子）在服务端。
  - ⚠️ **未定的张力**：甲方点名的 **Snurr / CompileFlow 是服务端 Java BPMN 引擎**，与「E2 在端上解释」是两条路。若确定走服务端 Java，端上只剩渲染 + 调接口——直接改范围与报价，**必须先跟甲方确认**。

## 4. 三块共用地基（编辑器与解释器都站在其上）

1. **算子注册表**：一条算子定义 = 类型化输入输出 + 端上实现或 RPC 标记 + 表单描述。面板与画布由它自动生成 →「可扩充算子」才便宜。
2. **表达式模型（可移植 AST，绝不 eval）**：操作数有来源（字面量 / 变量 / 组件数据 / 查询结果）与类型；比较器 / 函数是注册出来的。一处定义，三处复用（网关条件 / 算子入参 / UI 绑定）。
3. **精简 flow schema**：运行时真正消费的格式。

## 5. 图化编辑器（图化引擎产品化）

- 形态：BPMN 风格——开始事件、排他网关、任务节点（算子实例）、连线；左侧按类分组的算子面板；右侧节点属性 + 可视化条件构造器。
- **关键决策：编辑器格式与运行时格式解耦。** 编辑器可用 bpmn-js（画布 / 连线 / 撤销 / 序列化几乎白送），但**运行时只吃一份自己定的精简 flow JSON**——存盘时把图编译成 JSON。别让运行时去讲 BPMN。

## 6. 微信小程序可行性与风险

- **动态执行禁令**：解释器 = 代码（编进包）、流程 = 数据（JSON）→ 合规；`eval` / 远程下发 JS → 封杀。**流程在构建期编进包**，别向已发布 App 热推逻辑。
- **权限 / 合法域名**：解释器本身不需权限；网络算子只能打**预配置的 HTTPS 合法域名**，运行时不能加域名 → **RPC 只打 E3 固定域名，E3 再扇出到任意三方**（架构天然化解此限制）。
- **用户手势上下文**：支付 / getUserProfile 必须由点击同步触发；流程一旦 `await` 过手势就丢 → **手势算子不能排在异步算子之后**，需在 flow schema 层约束。
- **政策红线**：按项目编译 + 提交审核 = 安全；做一个「动态容器 App 运行时变任意应用」= 会被拒。
- **次要**：主包 2MB（用分包）；E2 频繁写状态触发 `setData` → 批量写；编辑器（bpmn-js）是设计态、不进手机包。

## 7. 第一期：走通式骨架（Phase 1）

- **纵切场景**：登录（界面 + 图化流程 + 登录算子）+ 商品列表（界面 + 请求算子含鉴权 + 商品列表算子 + 填充界面）。
- **为何选得准**：登录 + 带鉴权列表恰好把所有接缝各踩一遍——表单输入 → 流程 → RPC → 服务端逻辑 → token → 跨页状态 → 带鉴权再请求 → 查数据 → 列表渲染。
- **纪律**：引擎要「通用地薄」，不能写死。验金石——加第三个页面 / 算子时，**引擎代码零改动**，只加 flow JSON + 注册算子。
- **本切强制、不可推迟的两项**：错误处理（登录会失败，要错误分支）、状态作用域（token 跨页）。
- **可推迟**：数据建模器（先塞种子数据）、大部分算子广度、复杂控制流。
- **工期**：单资深全职 ~4–6 周（happy path ~2 周）；AI 辅助能压缩，但集成 / 设计 / 真机调试压不动。

## 8. 商务策略

- **量级**：完整平台七位数；第一期（骨架 + 可视化编辑器 + RisingWave/Snurr/CompileFlow）估 ¥12–25 万，**定 ¥20 万 / 3 个月**。
- **定价哲学**：给结果与风险定价，不是给工时；能力稀缺（多数程序员做不了渲染 / 图化引擎）；看**买家处境**（烧过几百万）而非卖家钱包；自由职业开价 ≈ 2× 受雇月薪。
- **AI 提速现实**（谈判自持，不外露）：GitHub/Accenture「最高 55%」仅限孤立样板任务；METR 2025 RCT 显示熟练开发者在真实复杂任务上**慢 19%、却自觉快 24%**。故承接方「一周做完」的自估是**最好情况**，且**无需向甲方透露**。
- **付款结构（红线）**：**绝不签「整体交付才付款」**。分期、每期独立交付独立收款、每期预付 + 尾款、每期小到「收不到也扛得住」。这同时满足甲方「做完才付」（指每期）与承接方「持续现金流」。
- **谈判**：¥20 万要有可信锚（逐条交付项撑住）；折扣**只换真承诺**（框架协议 / 第二期预定），别无条件白送；**不把总价锚在第一期**。
- **砍价杠杆（预算 < 20 万时按序砍）**：① RisingWave → 普通 Postgres；② 缓做小程序双端（第一期本就先 H5）；③ 减商城组件；④（最后才动）流程编辑器——它是卖点命根子，真要省退成「只读可视化」。**底线 ~¥13–14 万**仍保住三大卖点；再低就是 demo，要明说降级了什么。
