一、完整平台全景架构 Full Platform Architecture

低代码平台的本质，是「用一份声明式描述把 App 讲清楚（界面 / 数据 / 逻辑），再由引擎去解释并运行它」。完整平台由五层、三大引擎构成，横跨设计态、运行态、数据、集成与平台底座——这也是原 PC 版多年建设所沉淀的复杂度。

这张图代表整个工程的量级（多期推进、投入以百万计）。能把它讲清楚、并按期逐块落地，是本项目最稀缺的能力门槛——大多数团队卡在「界面拖拉」之后的渲染引擎与图化执行引擎上。

六、三大引擎内部机制 Engine Internals

前面五部分从架构全景、分期路线、交付范围的角度阐述了平台"是什么"和"交付什么"。本部分深入引擎内部，解释"怎么做"——这也是把 Schema 变成可运行应用的真正技术壁垒。大多数低代码团队止步于"界面拖拉"之后，正是因为无法跨越这三道引擎的门槛。

本节面向技术决策者。所有数据结构和算法均为第一期实现范围，与后续各期递增兼容。核心设计原则：Schema 是唯一真相源（Single Source of Truth）——设计态产出 Schema，运行态消耗 Schema，引擎之间通过 Schema 和 RPC 协议通信，不直接耦合。

6.1 共享基础设施：Schema · 算子 · 组件注册

三大引擎共享三块基础设施。Schema 定义了平台的"语言"，算子定义了平台的"能力"，组件注册定义了平台的"词汇"——引擎只负责"解释"。

6.1.1 Schema 体系 —— 平台的"语言"

Schema 是 JSON 格式的声明式描述，定义了组件树、流程、数据模型的结构。以下为第一期核心 Schema 类型：

组件树 Schema（ComponentTreeSchema）——定义页面结构 
// 页面 Schema —— 一个页面对应一个 Schema 文件
interface PageSchema {
  id: string;                     // 页面唯一标识
  version: string;                // Schema 版本号，向前兼容
  components: ComponentNode[];   // 根组件列表（支持多根）
  globalState: StateDefinition;   // 页面级状态定义
  flows: string[];               // 关联的流程 ID
}

interface ComponentNode {
  id: string;                    // 组件实例唯一 ID（用于 diff 的 key）
  type: string;                  // 组件类型名，从 Registry 查找
  props: Record<string, Expression>; // 属性：静态值或表达式
  children?: ComponentNode[];    // 子组件
  events?: EventBinding[];       // 事件绑定 → 触发流程
  condition?: Expression;        // 条件渲染：{ items.length > 0 }
  loop?: {                      // 列表渲染
    source: Expression;         // 数据源：{ state.productList }
    itemKey: string;           // diff 的 key 字段，如 "id"
    itemName: string;          // 迭代变量名，如 "item"
  };
  style?: Record<string, Expression>; // 动态样式
  slots?: Record<string, ComponentNode[]>; // 命名插槽
}

interface EventBinding {
  event: string;                 // 事件名：onClick / onChange / onLoad
  action: FlowAction;             // 触发动作
}

type FlowAction = {
  type: 'invoke_flow';         // 调用流程
  flowId: string;               // 目标流程 ID
  params?: Record<string, Expression>;
} | {
  type: 'set_state';           // 直接修改状态
  expression: Expression;
} | {
  type: 'navigate';            // 页面跳转
  target: string;
};
流程 Schema（FlowSchema）——定义业务流程 
// 流程 Schema —— 一个流程对应一个 Schema 文件
interface FlowSchema {
  id: string;
  name: string;
  version: string;
  nodes: FlowNode[];              // 节点列表
  edges: FlowEdge[];              // 边列表（连线）
  variables: VariableDefinition[]; // 流程内部变量
  timeout?: number;              // 流程超时（ms），默认 60s
  errorStrategy: 'stop' | 'continue' | 'compensate';
}

interface FlowNode {
  id: string;
  type: 'action' | 'condition' | 'loop' | 'parallel' | 'start' | 'end';
  label: string;                 // 节点显示名
  operator?: string;             // 算子名称（action 节点）
  config?: Record<string, any>;  // 算子配置
  inputMapping?: Record<string, Expression>; // 输入映射
  outputMapping?: Record<string, string>; // 输出映射到变量
  errorHandling?: {
    retry?: { maxAttempts: number; backoff: number }; // 重试
    fallbackNodeId?: string;     // 降级节点
    compensate?: string;         // 补偿算子名
  };
  timeout?: number;              // 节点超时
}

interface FlowEdge {
  id: string;
  from: string;                  // 来源节点 ID
  to: string;                    // 目标节点 ID
  condition?: Expression;        // 条件分支：{ result.code === 0 }
  label?: string;                // 边标签
  default?: boolean;             // 是否为默认分支
}

6.1.2 算子开发 —— 平台可扩展性的核心

大多数低代码平台的"可扩展性"止步于"能加组件"。本平台的真正扩展点在算子——一个算子就是流程中的一个可执行节点。给 100 个预置组件，不如给一个能让使用者注册任意算子的接口。

算子接口定义——实现此接口即可注册为新算子 
interface OperatorDefinition {
  name: string;                    // 算子唯一标识，在流程编辑器中按名引用
  label: string;                   // 显示名称，出现在算子面板中
  description: string;             // 功能描述
  category: 'data' | 'ui' | 'integration' | 'custom';
  inputSchema: JSONSchema;         // 输入参数 Schema → 编辑器自动生成配置表单
  outputSchema: JSONSchema;        // 输出 Schema → 下游节点可引用
  location: 'local' | 'remote';  // local = 端上执行，remote = RPC 到 E3 服务引擎
  handler: (input: Record<string, any>, ctx: ExecutionContext) => Promise<any>;
}
核心设计：分离「定义」与「消费」 
// 算子定义者只需关注一件事：实现 handler
const loginOperator: OperatorDefinition = {
  name: 'login',
  label: '用户登录',
  category: 'integration',
  inputSchema: {
    type: 'object',
    properties: { username: { type: 'string' }, password: { type: 'string' } },
  },
  location: 'remote',
  async handler(input, ctx) {
    const user = await db.verify(input.username, input.password);
    return { token: jwt.sign(user), user };
  },
};

// 注册后，流程编辑器自动出现"用户登录"算子
// inputSchema 自动生成配置表单（username 输入框 + password 输入框）
// outputSchema 自动提示下游节点可引用的字段（token、user）
// location='remote' 自动走 RPC 到 E3 服务引擎执行
operatorRegistry.register(loginOperator);
算子才是平台真正的可扩展性。组件扩展的是"界面能长什么样"，算子扩展的是"流程能做什么"。因为任何业务系统的核心逻辑都在流程中，不在界面上。一个平台预置多少算子不重要——重要的是注册新算子的成本有多低。本平台的设计目标是：加一个新算子只需实现一个接口、注册一行代码，编辑器自动适配、引擎自动路由。

6.1.3 组件注册表（Component Registry）

组件注册表是设计态组件库与运行态渲染引擎之间的桥梁。设计态列出可用组件，运行态按名查找渲染器。

interface ComponentDefinition {
  name: string;                    // 组件名，如 "Button", "ProductCard"
  propsSchema: JSONSchema;         // 属性 Schema → 设计态校验
  renderer: ComponentRenderer;     // 渲染函数
  defaultProps?: Record<string, any>;
  supportedPlatforms: ('h5' | 'miniprogram')[];
}

// 平台适配接口 —— 解耦引擎与具体平台
interface PlatformAdapter {
  createElement(type: string, props: any): NativeElement;
  setAttribute(el: NativeElement, key: string, value: any): void;
  appendChild(parent: NativeElement, child: NativeElement): void;
  removeChild(parent: NativeElement, child: NativeElement): void;
  addEventListener(el: NativeElement, event: string, handler: Function): void;
  setTextContent(el: NativeElement, text: string): void;
}

核心流程 1：Schema → Virtual Tree（首次渲染） 
// 伪代码：首次渲染页面
function buildVirtualTree(
  schema: ComponentNode[],
  registry: ComponentRegistry,
  ctx: EvaluationContext
): VirtualNode[] {
  return schema.map(node => {
    // 1. 条件渲染：先评估 condition
    if (node.condition && !evaluate(node.condition, ctx))
      return null; // 不渲染

    // 2. 从注册表查找组件定义
    const def = registry.resolve(node.type);

    // 3. 解析 props 中的表达式
    const resolvedProps = resolveProps(node.props, ctx);

    // 4. 列表渲染：展开 loop
    if (node.loop) {
      const items = evaluate(node.loop.source, ctx);
      return items.map(item => {
        const childCtx = ctx.extend({ [node.loop.itemName]: item });
        return new VirtualNode(node.id, def, resolvedProps,
          buildVirtualTree(node.children, registry, childCtx),
          node.events);
      });
    }

    return new VirtualNode(
      node.id, def, resolvedProps,
      buildVirtualTree(node.children, registry, ctx),
      node.events
    );
  }).filter(Boolean);
}
核心流程 2：Reconciler 增量更新 —— 为什么不能每次都全量渲染

当状态变化时（如用户点击、接口返回数据），如果全量重新渲染整个页面，性能将不可接受。Reconciler 通过 Tree Diff 算法只更新变化的部分：

// 状态变化 → 找出受影响的 VirtualNode → Diff → Patch → 批量提交
function reconcile(
  oldTree: VirtualNode,
  newTree: VirtualNode,
  patches: Patch[]
): void {
  // 优化 1: 类型不同 → 直接替换整棵子树
  if (oldTree.type !== newTree.type) {
    patches.push({ type: 'REPLACE', oldNode: oldTree, newNode: newTree });
    return;
  }

  // 优化 2: props 浅比较 → 只更新变化的属性
  const changedProps = diffProps(oldTree.props, newTree.props);
  if (Object.keys(changedProps).length > 0)
    patches.push({ type: 'UPDATE_PROPS', node: oldTree, props: changedProps });

  // 优化 3: 子节点 key-based diff → O(n) 而非 O(n^3)
  const oldChildren = oldTree.children || [];
  const newChildren = newTree.children || [];
  reconcileChildren(oldChildren, newChildren, patches);
}

// 批量提交：多个状态变更合并为一个渲染帧
let pendingPatches: Patch[] = [];
let rafId: number | null = null;

function scheduleUpdate(patch: Patch): void {
  pendingPatches.push(patch);
  if (!rafId)
    rafId = requestAnimationFrame(() => {
      commitPatches(pendingPatches); // 批量提交到平台适配层
      pendingPatches = [];
      rafId = null;
    });
}

核心流程 1：拓扑排序与合法性校验 
// 校验规则：在排序之前必须全部通过
function validateFlow(schema: FlowSchema): ValidationResult {
  const errors: string[] = [];

  // 规则 1: 必须有且只有一个 Start 节点
  const startNodes = schema.nodes.filter(n => n.type === 'start');
  if (startNodes.length !== 1)
    errors.push('流程必须有且只有一个 Start 节点');

  // 规则 2: 至少有一个 End 节点
  if (!schema.nodes.some(n => n.type === 'end'))
    errors.push('流程必须至少有一个 End 节点');

  // 规则 3: 所有节点必须可达（无孤立节点）
  const reachable = new Set<string>();
  function dfs(nodeId: string) {
    if (reachable.has(nodeId)) return;
    reachable.add(nodeId);
    schema.edges.filter(e => e.from === nodeId)
      .forEach(e => dfs(e.to));
  }
  dfs(startNodes[0].id);
  for (const node of schema.nodes)
    if (!reachable.has(node.id) && node.type !== 'start')
      errors.push(`节点 ${node.id} 不可达`);

  // 规则 4: 条件分支的每条边必须有 condition 或 default
  for (const node of schema.nodes.filter(n => n.type === 'condition')) {
    const outgoing = schema.edges.filter(e => e.from === node.id);
    if (!outgoing.some(e => e.default))
      errors.push(`条件节点 ${node.id} 缺少默认分支`);
  }

  return { valid: errors.length === 0, errors };
}

// Kahn 拓扑排序 —— 同时检测环
function topologicalSort(schema: FlowSchema): string[] {
  const inDegree = new Map<string, number>();
  const adj = new Map<string, string[]>();

  for (const node of schema.nodes) {
    inDegree.set(node.id, 0);
    adj.set(node.id, []);
  }

  for (const edge of schema.edges) {
    adj.get(edge.from)!.push(edge.to);
    inDegree.set(edge.to, (inDegree.get(edge.to) || 0) + 1);
  }

  const queue: string[] = [];
  for (const [id, deg] of inDegree)
    if (deg === 0) queue.push(id);

  const result: string[] = [];
  while (queue.length > 0) {
    const node = queue.shift()!;
    result.push(node);
    for (const neighbor of adj.get(node) || []) {
      inDegree.set(neighbor, inDegree.get(neighbor)! - 1);
      if (inDegree.get(neighbor) === 0) queue.push(neighbor);
    }
  }

  // 如果 result.length < nodes.length，说明存在环
  if (result.length !== schema.nodes.length)
    throw new CyclicFlowError('流程包含环，请检查连线');

  return result;
}
核心流程 2：状态机执行模型 —— 为什么不用 async/await？

流程执行必须可暂停、可序列化、可恢复、可调试。普通的 async/await 无法满足这些需求——一个 await 中的 Promise 无法被序列化保存。状态机方案天然支持这些能力：

// 状态机 —— 状态可随时 JSON.stringify 保存，下次恢复执行
interface FlowState {
  status: 'idle' | 'running' | 'paused' | 'completed' | 'failed';
  currentNodeId: string | null;
  context: {
    variables: Map<string, any>;       // 流程变量
    nodeOutputs: Map<string, any>;   // 每个节点的输出
    error: Error | null;               // 当前错误
  };
  callStack: string[];                  // 子流程调用栈
  history: TransitionRecord[];           // 状态转换历史（调试用）
}

/*
状态转换图：
  idle --start()--> running --await RPC--> paused --resume()--> running
                       |                                        |
                       +-- all nodes done --> completed         |
                       +-- error, no retry --> failed           |
                       failed --retry()--> running
*/

class FlowStateMachine {
  private state: FlowState;

  async step(): Promise<void> {
    if (this.state.status !== 'running') return;

    const node = this.getCurrentNode();
    const nextNodeId = await this.executeNode(node);

    // 记录转换历史
    this.state.history.push({
      from: this.state.currentNodeId,
      to: nextNodeId,
      timestamp: Date.now(),
      output: this.state.context.nodeOutputs.get(node.id),
    });

    if (!nextNodeId || this.isEndNode(nextNodeId))
      this.state.status = 'completed';
    else {
      this.state.currentNodeId = nextNodeId;
      await this.step(); // 递归执行下一步
    }
  }

  // 序列化：保存当前状态，稍后恢复
  serialize(): string {
    return JSON.stringify(this.state);
  }

  static deserialize(json: string): FlowStateMachine {
    const state = JSON.parse(json);
    return new FlowStateMachine(state);
  }
}
核心流程 3：并行执行（Fork-Join） 
/*
并行节点的执行模型：
          +----------+
          | Parallel |  <-- fork 节点
          +----+-----+
       +-------+-------+
       v       v       v
    [Node A] [Node B] [Node C]  <-- 三个独立子上下文
       |       |       |
       v       v       v
    [End A] [End B] [End C]
       |       |       |
       +-------+-------+
               v
          +----------+
          |   Join   |  <-- join 节点
          +----------+
*/

class ParallelExecutor {
  async execute(node: FlowNode, ctx: ExecutionContext): Promise<NodeOutput> {
    const branches = this.getBranches(node); // 从 fork 节点出发的每条路径

    const results = await Promise.allSettled(
      branches.map(branch =>
        this.executeBranch(branch, ctx.fork()) // fork() 创建独立子上下文
      )
    );

    const errors = results.filter(r => r.status === 'rejected');
    if (errors.length > 0)
      throw new ParallelExecutionError(errors); // 聚合所有分支错误

    // 合并各分支的输出
    return results.reduce((merged, r) =>
      ({ ...merged, ...(r as PromiseFulfilledResult<any>).value }),
      {});
  }
}
核心技术挑战 1：Saga 补偿模式（分布式事务的最终一致性）

当流程中的某个节点失败，而之前的节点已经产生副作用（如创建了订单、扣减了库存），需要逆序执行补偿来恢复。这不是真正的数据库事务回滚，而是业务层的 Saga 补偿：

/*
正常执行: A → B → C → D → End
失败回滚: A → B → C x (失败)
                  |
            compensate(C) → compensate(B) → compensate(A)
*/

async function executeWithCompensation(
  flow: FlowSchema, ctx: ExecutionContext
): Promise<void> {
  const executedNodes: FlowNode[] = []; // 记录已执行的节点

  try {
    for (const node of executionPlan) {
      await executeNode(node, ctx);
      executedNodes.push(node);
    }
  } catch (error) {
    // 逆序执行补偿（跳过没有定义 compensate 的节点）
    for (const node of executedNodes.reverse()) {
      if (node.errorHandling?.compensate) {
        try {
          await executeOperator(node.errorHandling.compensate, ctx);
        } catch (compError) {
          // 补偿本身也可能失败 → 记录并人工介入
          this.logCompensationFailure(node, compError);
        }
      }
    }
    throw error; // 补偿完成后仍然抛出原始错误
  }
}
核心技术挑战 2：表达式在流程中的多层上下文解析

流程中表达式的上下文有严格的优先级：

/*
上下文查找优先级（从高到低）：
1. 当前节点的 inputMapping 形参 → 如 { input.userId }
2. 前置节点的 output       → 通过 outputMapping 绑定
3. 流程变量 (flow variables) → 在 FlowSchema.variables 中定义
4. 页面状态 (page state)     → 来自 UI Layout Engine
5. 会话状态 (session state)  → token、userId 等
*/

class FlowExpressionContext implements EvaluationContext {
  resolve(name: string): unknown {
    // 1. 当前节点输入形参
    if (this.inputParams.has(name)) return this.inputParams.get(name);

    // 2. 前置节点输出
    if (this.nodeOutputs.has(name)) return this.nodeOutputs.get(name);

    // 3. 流程变量
    if (this.flowVariables.has(name)) return this.flowVariables.get(name);

    // 4. 页面状态
    if (this.pageState.has(name)) return this.pageState.get(name);

    // 5. 会话状态
    if (this.session.has(name)) return this.session.get(name);

    throw new ExpressionError(`未定义的变量: ${name}`);
  }
}

interface OperatorDefinition {
  name: string;
  version: string;
  description: string;
  inputSchema: JSONSchema;       // 输入校验 Schema
  outputSchema: JSONSchema;      // 输出校验 Schema
  handler: (input: any, ctx: ExecutionContext) => Promise<any>;
  config?: {
    timeout: number;             // 默认 30s
    retry: { maxAttempts: number; backoff: 'fixed' | 'exponential' };
    rateLimit?: { maxPerSecond: number };
    cache?: { ttl: number };     // 结果缓存
  };
}

class OperatorRegistry {
  private operators = new Map<string, OperatorDefinition[]>();

  register(def: OperatorDefinition): void {
    const versions = this.operators.get(def.name) || [];
    versions.push(def);
    this.operators.set(def.name, versions);
  }

  resolve(name: string, version?: string): OperatorDefinition {
    const versions = this.operators.get(name);
    if (!versions) throw new OperatorNotFoundError(name);
    if (version)
      return versions.find(v => v.version === version)
        || versions[versions.length - 1]; // 找不到指定版本 → fallback 到最新
    return versions[versions.length - 1]; // 返回最新版本
  }
}
鉴权流程 
/*
  登录请求（username + password）
    |
    v
  验证凭据（bcrypt.compare）
    | 成功
    v
  签发 JWT Token:
    { sub: userId, tenant: tid, exp: +7d, iat: now }
    |
    v
  返回 Token → 端上存入 Session
  后续请求带 Authorization: Bearer <token>
    |
    v
  Auth Middleware 校验：
    1. 提取 Authorization header
    2. 验证签名 & 过期时间
    3. 注入 userId / tenantId 到 context
*/

class AuthMiddleware {
  async handle(req: RpcRequest, next: () => Promise<RpcResponse>): Promise<RpcResponse> {
    // 白名单：登录算子不需要鉴权
    if (this.isPublicOperator(req.operator)) return next();

    const token = this.extractToken(req);
    if (!token) throw new AuthError('未提供认证令牌');

    const payload = await this.verifyToken(token); // 签名 & 过期校验
    if (!payload) throw new AuthError('令牌无效或已过期');

    // 注入用户信息到请求上下文
    req.context.userId = payload.sub;
    req.context.tenantId = payload.tenant;

    return next();
  }
}

七、微信小程序合规说明 Mini Program Compliance

甲方可能会关心一个问题：低代码平台生成的微信小程序，界面是根据配置文件运行时动态渲染的，还是编译期生成的？这直接关系到能否通过微信审核。以下给出明确的技术回答。

简短回答：编译期生成，不是运行时动态渲染。Schema（页面结构 + 流程定义）在构建阶段被编译进小程序代码包，随代码一起提交微信审核。审核通过后，小程序的行为由已审核的代码 + 已审核的 Schema共同决定。运行时只从服务器拉取业务数据（如商品列表），不拉取页面结构或流程逻辑。

解释器代码（渲染引擎 + 流程引擎）构建期✗ 不可改随代码包审核
Schema（页面结构 + 流程定义）构建期✗ 不可改随代码包审核
业务数据（商品列表、用户信息）运行期✓ 可动态变化不审核（数据内容）

7.2 与微信审核红线的对照

微信《小程序运营规范》中与本方案相关的条款：
微信红线本方案应对结论
禁止动态下发可执行代码（eval / 远程 JS） 解释器是编译进包的原生 TypeScript 代码，不使用 eval。流程是 JSON 数据，由解释器解释执行，不是 eval 执行 ✓ 合规
禁止「动态容器」—— 一个 App 运行时变成任意应用 Schema 在构建期编译进包，不从服务器下发。小程序上线后，页面结构不能热更新。如需改变结构，必须重新构建并提交审核 ✓ 合规
网络请求只能访问预配置的合法域名 端上只通过 RPC 访问 E3 服务引擎的固定域名（已配置在 request 合法域名列表中）。E3 再扇出到第三方，端上不直接访问任意外部域名 ✓ 合规
支付 / 获取用户信息必须由用户点击同步触发 在流程 Schema 层约束：手势算子（支付 / getUserProfile）不能排在异步算子之后。编辑器在保存时校验此规则 ✓ 合规

7.3 技术实现：Schema 如何编译进小程序

// 构建流程：Schema 在构建时被打包进小程序，不是运行时拉取

// 步骤 1: Studio 导出项目时，序列化 Schema 为 JSON 文件
// 输出: project.json（包含 pages / flows / theme）

// 步骤 2: Taro 构建时，将 project.json 作为静态资源打包
import projectSchema from './project.json';
import { validateProject } from '@functorz/schema';

// 步骤 3: 启动时校验并加载（不是 fetch，是 import）
const project = validateProject(projectSchema);

// 步骤 4: 解释器根据 Schema 渲染界面
// 这一切发生在已审核的代码包内，不依赖服务器下发任何逻辑
renderPage(project.pages[0]);

// 运行时只拉取业务数据（商品列表等），不拉取界面结构
const products = await E3.call('getProductList');
总结：本方案的小程序架构与微信审核政策完全兼容。核心原则是「代码 + Schema 编译进包、随包审核；运行时只拉数据、不拉结构」。这与一个普通 Taro 小程序的行为模式完全一致——区别仅在于，普通小程序的界面是手写 JSX 编译的，低代码小程序的界面是 Schema 驱动的渲染引擎编译的。两者都经过微信审核，都不存在"绕过审查"的可能。
本建议书仅覆盖第一期。完整平台为多期框架合同，各期独立范围、独立验收、独立计价。
